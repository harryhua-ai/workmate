# 1. 项目概述

## 1.1 项目名称

亚马逊 SP 广告智能运营助手 MVP

## 1.2 项目目标

本项目拟建设一套围绕单个 Listing / 广告对象持续运行的 Agent 工作系统，聚焦亚马逊 SP 广告场景，帮助客户先完成一个可真实使用的 MVP 闭环。

本期目标不是一次性固化完整广告策略，而是优先交付一个可运行、可协作、可推送、可执行基础动作的最小可用系统，覆盖从投前检查到日常运营的核心链路。

## 1.3 一句话定义

这不是一个静态报表工具，而是一套围绕 Amazon SP 广告长期运行的 Thread + Run 式智能运营工作台。

# 2. 项目背景与建设价值

当前亚马逊广告运营存在几个典型问题：

> • 数据分散在广告、Listing、库存、关键词、排名等多个系统中，人工汇总成本高
>
> • 投前判断、词库准备、广告搭建、巡检复盘之间缺少统一工作流
>
> • 日常运营高度依赖人工盯盘，异常发现不及时
>
> • 报告与动作脱节，很多结论无法直接沉淀为可执行任务
>
> • 团队经验难复用，难以沉淀为长期可进化的运营能力

本项目希望通过 Agent + MCP + 自动化调度的方式，把高频 SP 广告运营流程沉淀为一个可持续运行的产品原型，帮助客户先建立：

> • 一条统一的运营入口
>
> • 一套稳定的结构化工作单元
>
> • 一组可复用的广告运营基础能力
>
> • 一种"系统主动推送 + 用户任务协作 + 必要时直接执行"的工作方式

## 2.1 项目基础条件与交付前提

本项目具备较好的落地基础，原因在于：

> • 我方已具备亚马逊开发者身份，并已拥有相应 API 接入能力
>
> • 广告、业务数据、调度与通知能力可基于现有接口体系进行集成
>
> • 本项目默认采用企业私有化部署方式实施，不走公有化共享数据架构
>
> • 核心业务数据、账号权限、运行日志均存放于客户可控环境内，降低数据外泄风险

这意味着，本项目并不是从零开始验证"能否接 API"，而是在已有接口条件下，围绕业务场景完成产品化封装与交付。

## 2.2 私有化部署与成本说明

本项目默认采用企业私有化部署方式，核心原则如下：

> • 系统部署在客户指定服务器或客户可控云环境中
>
> • 业务数据不对外分发，不进入无关第三方存储链路
>
> • Access Token、密钥、运行配置按企业级方式独立管理
>
> • 系统运行所需的 Token 与服务器资源主要用于日常巡检、周复盘、消息推送与基础查询

基于 MVP 阶段的能力边界判断，本项目整体 Token 成本与服务器成本开销可控，且通常较小，不会形成与项目主体投入不成比例的长期基础设施负担。

# 4. 目标产品形态

## 4.1 交互形态

MVP 阶段建议采用"飞书内对话 + 结构化 Run 结果沉淀 + 飞书文档留档"的产品形态，以最短路径验证客户是否真实使用。

## 4.2 交互入口

围绕"入站 + 出站"两个入口组织：

\#### 入站入口 用户从飞书 Thread 中发起：

> • 任务请求
>
> • 自定义查询
>
> • 审批确认
>
> • 直接动作
>
> • 补充信息

\#### 出站入口 系统主动向用户推送：

> • 日巡检摘要
>
> • 周复盘摘要
>
> • 异常告警
>
> • 执行动作回执
>
> • 待确认事项
>
> • 待人工处理事项

## 4.3 核心工作模型

系统采用 **Thread + Run** 模型：

> • \`Thread\` 是唯一输入入口，也是长期协作主线
>
> • \`Run\` 是一次具体工作的结构化执行单元
>
> • 所有检查、方案、巡检、复盘、执行动作均形成独立 Run
>
> • Run 摘要回流到 Thread
>
> • 用户对 Run 的追问和后续动作统一回到 Thread 中完成

# 5. 核心用例范围

## UC-01 投前检查

输出可投 / 暂缓投放结论、风险清单、待处理事项。

## UC-02 关键词词库初始化

输出首轮关键词库、核心词、长尾词、否定词候选与优先级建议。

## UC-03 SP 搭建方案生成

输出 Campaign / Ad Group / Keyword 结构建议，以及预算和竞价建议。

## UC-04 SP 发布执行

在用户确认后将方案发布到广告账户，并返回发布结果与日志。

## UC-05 天粒度巡检

每天自动识别广告异常，输出优化建议和日报摘要。

## UC-06 周粒度复盘

按周汇总表现，输出问题归因、下周建议和人工操作汇总。

## UC-07 主动推送通知

在巡检、复盘、异常发现、动作执行后主动推送结果。

## UC-08 任务式协作

支持用户在长期 Thread 中发起明确任务并获取结构化结果。

## UC-09 自定义查询

支持用户按问题查询状态、原因、建议和对象级结论。

## UC-10 直接动作

支持暂停 Campaign、暂停关键词、调整预算、添加否定词等基础动作。

## UC-11 审计与人工操作记录

记录变更动作、执行主体、时间与结果，形成可回溯日志。

## UC-12 Skill 自进化

支持 Skill、记忆和规则的持续沉淀与升级，但在本期以候选沉淀和人工审核机制为主，不做完全自动黑盒升级。

# 6. 系统能力与外部依赖

## 6.1 必需能力

本项目将围绕以下 MCP / 外部能力进行集成：

> • Amazon Ads MCP：广告读取、结构创建、动作执行
>
> • Seller / Business Data MCP：Listing、库存、价格、评论等基础数据读取
>
> • Keyword Intelligence MCP：外部关键词与搜索量参考
>
> • Organic Rank MCP：关键词自然排名及趋势
>
> • Scheduler / Automation MCP：定时巡检、周复盘、例行任务
>
> • Notification MCP：日报、周报、异常提醒、执行回执推送
>
> • Report Storage / Data Layer：Run、报告、日志、人工记录存储

## 6.2 降级策略

如部分能力暂未就绪，系统支持以下降级运行：

> • 缺关键词能力时，仅基于产品信息与站内词生成基础词库
>
> • 缺自然排名能力时，仅输出广告指标巡检与复盘
>
> • 缺广告执行权限时，仅生成建议，不执行动作
>
> • 缺通知能力时，仅在系统内查看，不主动推送
>
> • 缺历史存储能力时，仅保留即时结果，弱化回溯能力

## 6.3 系统实现分层

为便于客户理解项目交付边界，本系统实现分为两层：

\#### A. 软件逻辑实现 软件逻辑层负责把系统真正运行起来，重点是"系统能力"和"工作流闭环"，主要包括：

> • Thread + Run 工作流机制
>
> • Amazon Ads / Seller / Keyword / Rank 等接口接入
>
> • 调度、通知、日志、存储等底座能力
>
> • 任务发起、查询、执行、回执与状态流转
>
> • 权限控制、异常处理、降级运行与部署运维

这一层的核心价值是确保系统稳定可用、流程可跑通、结果可追溯，是整个产品的基础设施层。

\#### B. 业务策略 Skill 实现 Skill 层负责把"业务经验"沉淀成系统可调用的策略能力，重点是"怎么判断、怎么建议、怎么解释"，主要包括：

> • 投前检查的业务判断逻辑
>
> • 关键词词库初始化策略
>
> • SP 搭建结构与预算、竞价建议逻辑
>
> • 日巡检的异常识别与归因逻辑
>
> • 周复盘的问题归因与下周建议逻辑
>
> • 日报、周报、异常提醒的话术与结构模板
>
> • 用户查询时的解释口径与动作建议

这一层更偏业务方法论沉淀，不是单纯代码实现，而是把运营策略逐步产品化、结构化和可复用化。

## 6.4 Skill 的自进化机制

本系统支持 Skill 的持续进化，但在 MVP 阶段以"可沉淀、可审核、可逐步放大"的方式落地，不追求完全自动黑盒升级。

Skill 的自进化主要包括三类路径：

> • 用户主动声明：用户明确提出"以后遇到这类情况按这个逻辑处理"，系统可将其沉淀为候选 Skill 规则
>
> • Agent 后台复盘：系统在巡检、复盘和任务执行后，对高频有效逻辑进行总结，形成候选 Skill
>
> • 开发人员提炼：开发人员从运行记录、记忆和用户自定义规则中提炼稳定策略，升级为正式 Skill

因此，本项目交付的不只是一个固定逻辑系统，而是一套"软件底座 + Skill 策略层"共同演进的 Agent 工作系统。
