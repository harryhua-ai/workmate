| 项目要素   | 内容                                                                                              |
| ---------- | ------------------------------------------------------------------------------------------------- |
| 产品定位   | 围绕单个 Listing / 广告对象持续运行的 Amazon SP 广告智能运营助手（Thread + Run 式智能运营工作台） |

## 一、产品能力与交付范围

### 1.1 MVP 交付的 12 个用例（UC）

原方案定义的 12 个核心用例：

| UC    | 用例名称           | 说明                                  | 交付概率 |
| ----- | ------------------ | ------------------------------------- | -------- |
| UC-01 | 投前检查           | 分析 Listing 质量、竞品格局、类目机会 | 95%      |
| UC-02 | 关键词词库初始化   | 建立初始关键词库                      | 90%      |
| UC-03 | SP 搭建方案生成    | 输出广告结构建议                      | 90%      |
| UC-04 | SP 发布执行        | 执行广告发布                          | 80%      |
| UC-05 | 天粒度巡检         | 每日自动巡检广告表现                  | 95%      |
| UC-06 | 周粒度复盘         | 每周汇总表现变化，回顾异常            | 90%      |
| UC-07 | 主动推送通知       | 小时级异常发现与归因，主动推送        | 99%      |
| UC-08 | 任务式协作         | 基于 Thread 的多轮对话协作            | 90%      |
| UC-09 | 自定义查询         | 自然语言查询广告数据                  | 85%      |
| UC-10 | 直接动作           | 基础直接动作执行                      | 75%      |
| UC-11 | 审计与人工操作记录 | 所有操作可追溯、可审计                | 99%      |
| UC-12 | Skill 自进化       | Skill 策略层接入与管理机制            | 60%      |

**预期交付：** 6 周内交付 9-10 个核心 UC，概率 85-90%。

### 1.2 日常使用体验

**Day 1 — 首次接入：**

- 飞书 Thread 对话入口就绪
- 可发起投前检查、自定义查询等任务
- Agent 返回结构化 Run 结果

**Day 2-7 — 日常巡检：**

- 系统每日自动巡检广告表现
- 异常告警、日报摘要主动推送到飞书
- 小时级异常发现与归因

**Week 2+ — 深度协作：**

- 基于 Thread 的多轮任务协作
- 周复盘自动生成并推送到飞书文档
- 基础直接动作可执行（含确认流程）

**数据沉淀：**

- 结构化 Run 结果沉淀在飞书对话中
- 飞书文档留档，长期资产可追溯

| ---------------- | ---------------------------------------------------------- | -------------------------------- |

| **外部接口依赖** | 外部接口质量、权限、稳定性会直接影响部分能力上线深度       | Week 1 前置验证 Amazon Ads API   |
| **Skill 自进化** | 本期以"候选沉淀 + 人工审核 + 持续升级"为主                 | UC-12 降级为 Git + 人工审核      |

### 4.2 技术风险（OpenClaw 平台相关）

| 风险                  | 影响                           | 数能觉醒的应对                           |
| --------------------- | ------------------------------ | ---------------------------------------- |
| Amazon Ads API 稳定性 | API 限流、数据延迟可能影响功能 | Week 1 完成接口联调验证                  |
| AI 决策质量           | 建议不准确可能导致广告浪费     | 写操作需人工确认，变更影响预览，回滚能力 |
| 飞书卡片展示限制      | 部分复杂数据可能无法完美展示   | Week 1 提前验证，必要时降级              |
| OpenClaw 平台依赖     | 平台升级可能影响兼容性         | 关注版本更新，关键功能提供降级方案       |

## 一、技术架构

### 1.1 架构概览（基于 OpenClaw 平台）

```
┌─────────────────────────────────┐
│  OpenClaw Platform（平台提供）     │
│  ┌───────────────────────────┐  │
│  │ Gateway + Agent Loop       │  │  ← Agent 调度 + 会话管理
│  │ 飞书 Channel Plugin        │  │  ← 飞书 WebSocket/Cards/Drive
│  │ Session Store + 日志审计   │  │  ← 数据持久化
│  │ Cron + Queue               │  │  ← 定时调度
│  │ Multi-Agent Routing        │  │  ← 多租户隔离
│  └───────────────────────────┘  │
│         ↕ Plugin SDK             │
│  ┌──────────┬──────────────┐   │
│  │ Amazon   │ Seller Data  │   │  ← 数能觉醒开发
│  │ Ads Tool │ Tool         │   │
│  └──────────┴──────────────┘   │
│  ┌──────────────────────────┐  │
│  │ SP 广告 Skills (SKILL.md) │  │  ← 数能觉醒开发
│  └──────────────────────────┘  │
└─────────────────────────────────┘
         ↕
  Amazon Ads API / SP API
```

### 1.2 原方案 7 个 MCP 的处理方式

| 原方案 MCP                | OpenClaw 上的实现             | 数能觉醒工作量   |
| ------------------------- | ----------------------------- | ---------------- |
| Amazon Ads MCP            | **Plugin Tool**               | 需开发（2-3 周） |
| Seller/Business Data MCP  | **Plugin Tool**               | 需开发（1-2 周） |
| Keyword Intelligence MCP  | **合并到 Seller Data Plugin** | 需开发（简化）   |

| Scheduler/Automation MCP  | **Gateway Cron（平台提供）**  | 仅配置           |
| Notification MCP          | **飞书 Plugin（平台提供）**   | 仅配置           |

### 1.3 技术选型

| 组件       | 选型                            | 理由                                        |
| ---------- | ------------------------------- | ------------------------------------------- |
| 开发语言   | TypeScript (ESM)                | OpenClaw Plugin SDK 原生语言                |
| 插件框架   | OpenClaw Plugin SDK             | `api.registerTool()` / `api.registerHook()` |
| 安全拦截   | `before_tool_call` hook         | 写操作拦截                                  |
| 定时调度   | Gateway cron 配置               | 巡检/复盘触发                               |
| 数据持久化 | OpenClaw Session Store          | 不需要自建数据库                            |
| 消息通道   | 飞书（MVP）                     | OpenClaw Feishu Plugin 原生支持             |
| LLM 模型   | Sonnet（分析）+ Haiku（预处理） | 分级模型降低成本                            |
| 部署       | Docker 容器化一键部署           | OpenClaw 内置支持                           |

## 四、产品化与规模化

### 4.1 产品化原则

**所有客户差异通过配置而非代码实现。**

| 定制维度     | 实现方式       |
| ------------ | -------------- |
| ACoS 阈值    | Skill 配置参数 |
| 推送频率     | Cron 配置      |
| 广告策略     | SKILL.md 模板  |
| Listing 映射 | 客户配置文件   |

### 4.2 增长路径

```
Month 1-2: 首客户交付
  → 6 周完成 MVP
  → 验证端到端流程
  → 沉淀配置模板

Month 3-4: 第 2-3 客户
  → 1-2 周快速交付
  → 基于反馈优化

Month 5-8: 规模化
  → 目标 5-10 个付费客户
  → 上架 OpenClaw 插件目录

Month 9-12: 深耕
  → 推出 Enterprise 层
  → 探索按效果付费
```

### 4.3 收入预测（保守场景）

| 时间   | 客户数 | 年收入       | 说明            |
| ------ | ------ | ------------ | --------------- |
| Year 1 | 10-15  | 120-180 万   | 首单 + 后续客户 |
| Year 2 | 30-50  | 360-600 万   | 复制加速        |

### 4.4 竞争壁垒

2. **策略资产沉淀**（强）— 使用时间越长，策略库越值钱
3. **全通道覆盖**（强）— 20+ IM 通道即插即用
4. **先发优势**（中）— 第一个 OpenClaw SP 广告插件
5. **私有化部署**（中）— 数据安全可控

## 六、关键执行建议

### 6.1 第一优先级

1. **第 1 周前置验证 Amazon Ads API**
   - 用 2-3 天完成核心接口联调
   - 确认限流策略、数据延迟、字段完整性
   - 如有硬伤，立即与客户沟通

2. **AI 决策质量投入 ≥ 30% 总工期**
   - Prompt 工程、Few-shot 示例、边界情况处理
   - 基于真实数据迭代调优
   - 功能做出来但建议不靠谱，比功能少但建议精准更致命

### 6.2 技术执行要点

1. **Week 1 并行启动**：Plugin 开发和 Skill Prompt 编写同时进行
2. **安全优先**：写操作通过 `before_tool_call` hook 拦截
3. **Prompt 版本管理**：所有 SKILL.md 用 Git 管理
4. **模型分级**：Haiku 做预处理，Sonnet 做复杂分析

### 6.3 产品执行要点

1. **Thread + Run 模型**：保持对话连续性，结构化结果沉淀
2. **飞书文档留档**：长期资产可追溯
3. **不要过度承诺**：本期重点是"建议 + 基础执行 + 协作闭环"

### 6.4 商业执行要点

1. **从第一天按产品标准开发**：客户差异走配置不走代码
2. **第二个客户交付目标**：压缩到 1-2 周
3. **3 个月内验证 PMF**：确认产品-市场匹配
